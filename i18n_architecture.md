# Архитектура Интернационализации (i18n) Sunberg

В этом документе описан принятый архитектурный стандарт для перевода платформы Sunberg на поддержку множества стран и языков (Бельгия, Испания и др.).

## 1. Общая концепция: Config-Driven SPA с локализованными путями
Проект остается единым Single Page Application (SPA), развернутым на основном домене (например, `.be`). Логика стран и доступных языков управляется через параметры URL и файлы конфигураций.

**Структура URL:**
- `/:region/:lang/`
- Примеры: `/be/nl/`, `/be/fr/`, `/es/es/`, `/es/en/`.

## 2. Разделение зон ответственности (Separation of Concerns)

Мы строго разделяем инфраструктуру, переводы и бизнес-логику:

### 2.1 Инфраструктура (Роутинг и SEO)
- **Маршрутизация**: Управляется через `react-router-dom` в слое `App.jsx`. Роутер не знает о "фичах" стран, он только парсит параметры `/:region/:lang/` из URL.
- **SEO**: Управляется на уровне компонентов страниц (через `Helmet`). Мета-теги (`<title>`, `<meta>`, `hreflang`) генерируются динамически.

### 2.2 Переводы (Языки)
- **Инструмент**: Управляется **исключительно** через `react-i18next`.
- **Логика**: Компоненты ничего не знают о странах, они просто запрашивают переводы по ключам (например, `t('hero.title')`). 

### 2.3 Бизнес-логика (Конфигурация регионов)
- **Инструмент**: Файлы конфигурации в `shared/config/regions/` (например, `belgium.js`, `spain.js`).
- **Суть**: Конфиги содержат только Feature Flags (включить/отключить блок, тип оффера). Они **не управляют** роутингом или переводами.

## 3. Правила работы со слоем Pages и Widgets (FSD)

### Слой Pages (Композиция)
Страницы выступают "глупыми сборщиками". Только они имеют право запрашивать конфиг региона.
Страница принимает решение, рендерить виджет или нет, на основе конфига:
```jsx
// Пример внутри HomePage.jsx
{config.features.showSubsidies && <SubsidiesWidget />}
```

### Слой Widgets (Изоляция)
Виджеты **максимально изолированы**. Внутри виджета запрещено использовать проверки на страну (никаких `if (region === 'es')`).

## 4. Стратегия модификации блоков

Если блок для другой страны нужно немного изменить, мы используем два подхода:

### 🟢 Модификаторы (Пропсы)
Используются, когда изменения касаются **только визуальной части (CSS)** (отступы, цвета, небольшие изменения верстки).
- Виджет принимает нейтральный пропс, например `layoutVariant="compact"`.
- Страница передает этот пропс на основе конфига.
- Виджет остается универсальным и может переиспользоваться.

### 🔴 Создание нового виджета
Используется, когда меняется **внутренняя структура (DOM) или бизнес-логика**.
- Если в Испании в блоке появляется форма захвата лидов, а в Бельгии ее нет — мы **не усложняем** старый виджет.
- Мы создаем `WidgetSpain` (или `WidgetWithForm`) и подменяем его на уровне страницы:
```jsx
// На уровне страницы:
{config.features.useAlternativeHero ? <HeroSpain /> : <HeroClassic />}
```

## 5. Микроязыковая Архитектура Регионов

Мы реализовали гибкую структуру региональных конфигураций и маршрутизации:

- **Бельгия (`/be/`)**:
  - Язык по умолчанию: **Голландский (`nl`)**.
  - Поддерживаемые языки: `['nl', 'en', 'fr']`.
- **Испания (`/es/`)**:
  - Язык по умолчанию: **Испанский (`es`)**.
  - Поддерживаемые языки: `['es', 'en']`.

Маршрутизатор в `App.jsx` автоматически перенаправляет пользователя без указания региона на дефолтный регион и язык: `Navigate to="/be/nl"`.

## 6. Механизмы Fallback и Инициализация (i18next)

1. **Роутер как источник правды**: Параметры URL (`:region/:lang`) имеют наивысший приоритет. При переходе по путям, `RegionProvider.jsx` перехватывает `lang` и обновляет язык в `i18next` через `i18n.changeLanguage(lang)`.
2. **Безопасное переключение (Language Switcher)**: Логика в `Header.jsx` динамически генерирует пути переключения языков внутри текущего региона.
3. **Общие переводы и fallback**: Если ключ перевода отсутствует в текущем выбранном языке, `i18next` автоматически использует английский язык в качестве запасного (`fallbackLng: 'en'`).

## 7. Локализация Блоков и Сквозных Компонентов

Каждая страница и её компоненты полностью привязаны к конфигурации `i18n`. Переводы разбиты по объектам, соответствующим FSD-слоям страниц и виджетов:
- `home`: Блоки главной страницы.
- `aboutPage`: Виджеты страницы «О нас».
- `servicesPage`: Виджеты страницы услуг.
- `projectsPage`: Виджеты портфолио.
- `techPage`: Виджеты страницы технологий.
- `contactPage`: Виджеты и формы обратной связи.
- `header` / `footer`: Общие сквозные элементы меню и футера.
